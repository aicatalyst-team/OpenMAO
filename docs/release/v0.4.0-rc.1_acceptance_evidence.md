# v0.4.0-rc.1 Acceptance Evidence

This document maps the first real side-effecting provider release to code and test evidence.
It does not replace maintainer release approval or independent review.

**Status:** release candidate
**Last updated:** 2026-05-29
**Runtime:** TypeScript on Node.js 22 LTS

## Summary

v0.4.0 makes one real, side-effecting capability enforceable end to end through an OpenMAO-managed
provider and credential broker: a GitHub issue-comment provider. It validates the enforced-capability
boundary against a genuine network side effect while keeping the default demo deterministic and
credential-free.

## At-most-once semantics

OpenMAO guarantees at most one provider invocation per capability call. It does not guarantee remote
exactly-once: a request that times out after GitHub created the comment but before OpenMAO records
success is recorded as failed even though the comment exists. Operators reconcile such cases by
inspecting the issue's comments before retrying rather than re-issuing blindly.

## Upgrade note

This release adds `Capability.credential_handle` and enforces capability-bound credentials. A local
`.openmao` database created before v0.4.0 holding a credential-requiring capability without the new
field will block such calls; reset local state (`rm -rf .openmao`) or re-register the capability when
upgrading. The default demos already reset local state.

### Review findings and disposition

Fixed in this release candidate:

- `CapabilityCall.credential_handle` is constrained to a `cred_*` shape in both the TypeScript
  contract and the generated JSON Schema, matching the runtime enforcement (the call contract
  previously accepted any string).
- The GitHub provider reports a parse-specific error when a 2xx response body cannot be read, instead
  of a misleading "no response" message.
- The credential broker trims a resolved secret, so a whitespace-padded environment value is not
  forwarded verbatim.
- A contract test pins the `credential_handle` pattern in the generated schema.

Accepted limitations (documented; not reachable in the supported single-process, single-file
deployment):

- **Cross-process duplicate execution.** If two processes share one database and a duplicate resume
  arrives while the first provider call is still in flight, the duplicate can record a failed result
  for a side effect that ultimately succeeds. The durable guard still prevents a second provider
  invocation; reconcile per the at-most-once note above. In-process duplicates are joined correctly.
- **Execution-claim auditing.** The node effect that guards at-most-once execution is durable state
  but is not itself an event; the capability request and its result are both evented, and an orphaned
  claim is reconciled on the next resume. Emitting an explicit execution-claim event is a candidate
  follow-up.
- **Environment handle shape.** An environment-backed credential handle must use a lowercase
  `cred_[a-z0-9_]+` form; broader handle shapes are valid in the contract and for in-memory brokers
  but are rejected by the environment broker, clearly, at resolution time.
